The Sorcerer's Cave

Design & Requirements Specification — a language-agnostic blueprint for re-implementation
Solitaire adaptation of Terence Donnelly's 1978 board game · reverse-engineered from the VAX MACRO-32 reference build
Version 1.0 · 2026-06-12 · Primary audience: an AI agent (Claude Code) re-implementing the game in another language and/or a web browser

This document is a complete, self-contained specification of The Sorcerer's Cave as realised in the VAX MACRO-32 reference implementation under src/macro32/sorcerer/. It is written so that a competent engineer — or an AI coding agent — can re-implement the game faithfully without reading the MACRO-32 source, in any language, for a terminal or a browser.

It does three things at once:

  1. Weaves in the original 1978 rules (quoted in tan boxes) so the design rationale is visible.
  2. Specifies the implemented behaviour precisely — data schemas, algorithms, constants, and edge cases — resolving the solitaire ambiguities the rulebook leaves open (green boxes).
  3. Catalogues the gaps between the original manual and the current build (red boxes), each specified in enough detail to be implemented from scratch.
Display philosophy. All layout is described in logical units — panels, regions, and a glyph vocabulary — never in fixed terminal rows/columns. The reference build targets an 80×24 VT terminal via DEC's SMG$ screen library, but nothing here depends on that. A browser re-implementation should map each logical region to a DOM container and each glyph to a styled cell or sprite.

1. Game Overview & Objective

In the heart of a forest in a faraway land is the entrance to a vast underground labyrinth, the treasure-house of an evil Sorcerer… To protect his hoard, the Sorcerer has made by his magic arts an ever-changing Cave of many tunnels and chambers, and filled it with all manner of pitfalls and fearsome creatures.

You lead a small party of creatures down into a procedurally-revealed cave of many levels. Each turn you explore one area. You fight monsters, befriend strangers, gather treasure, and survive hazards, then try to climb back out alive through the Gateway with as high a score as possible.

The board game supports 2–4 players plus solitaire. This implementation is solitaire only. All player-interaction rules (trading, party-vs-party fights, unions, division of a party, zombies) are out of scope by design. The one solitaire-specific rule that matters is quoted here and specified in §8:

In solitaire play there is only one exploring party, and all rules governing turns and player interaction are ignored. Strangers which remain indifferent after three rolls of the die stay indifferent for the rest of the game.

Win / lose conditions

  • Escape — climb the Gateway stair on level 1 to leave the cave. You keep all points you carry.
  • Death — the entire party is killed (combat or hazard). Score is zero.
  • Quit — abandon the cave voluntarily. Scored on what you carry (but see §13: the reference build zeroes the breakdown on quit).

2. High-Level Architecture

↑ top

The game is a single-threaded, turn-based, blocking-input state machine over a body of shared mutable state. There is no real-time loop, no animation, and no concurrency. Each turn is driven entirely by one keystroke. This makes the design trivially portable: a browser version is an event loop where each key event advances the same state machine.

2.2 Top-level state machine

Game-state variable (GS) drives termination: 0=PLAYING, 1=ESCAPED, 2=DEAD, 3=QUIT. The turn loop checks GS at the top of every iteration and exits to scoring when non-zero.

3. Core Data Model

↑ top

All entities are small integer records in parallel arrays indexed by a stable integer id. Re-implementations may use objects/structs instead; the ids and field values below are normative.

3.1 Area cards (the large pack)

There are 61 area cards. Each is a tunnel, a chamber, or a special area, with 2–4 doorways and optionally a stair up and/or down. Each card is encoded as a single integer (bit-field):

bit 0  (1)    North exit
bit 1  (2)    East  exit
bit 2  (4)    South exit
bit 3  (8)    West  exit
bit 4  (16)   Chamber flag      (if clear → tunnel/corridor)
bit 5  (32)   Stair up
bit 6  (64)   Stair down
bits 7-9      Special type × 128:
                0 none   1 Gateway   2 Deep Pool   3 Viper Pit
                4 Tomb of Kings   5 Great Hall

Worked example - the Gateway card (value 175):
   NSEWU = N(1)+E(2)+S(4)+W(8)+stairUp(32) = 47
   special type 1 (Gateway) = 1 × 128 = 128
   47 + 128 = 175

The full 61-card table (string, special, encoded value, index) is in Appendix A. Card index 21 is the Gateway; it is removed from the pack and placed as the starting area, leaving 60 shuffled cards in the large pack.

Several special areas appear exactly once in the deck (Gateway, Deep Pool, Viper Pit, Tomb of Kings, Great Hall). They are distinguished on cards by the special-type bits, not by a separate list. The Tomb and Great Hall are ordinary chambers whose only distinction is that they cause extra small-card draws (§7).

3.4 Hazards

5 hazard types, ids 0–4: 0 Mutiny, 1 Trap, 2 Earthquake, 3 Medusa, 4 Ghouls. Behaviour and resolution order are in §7.2.

3.5 The small pack

The chamber deck (“small pack”) has 52 cards. Each card is encoded as one integer: 100+creatureId, 200+treasureId, or 300+hazardId. Composition:

Creatures (19)
  • 1× W-Hero
  • 3× Ogre
  • 2× Troll
  • 3× Wizard
  • 3× Spectre
  • 3× Dragon
  • 1× Sorcerer
  • 2× Giant
  • 1× Unicorn
Treasures (27)
  • 6× Silver, 6× Gold, 3× Gems
  • 1× each of the 12 artifacts (ids 3–14)
Hazards (6)
  • 1× Mutiny, 2× Trap, 1× Earthquake, 1× Medusa, 1× Ghouls
The original physical pack contains blank cards and a Sibyl that are removed before play, and (in some print runs) 3 cave Priests bringing the count to 55. This implementation uses the canonical 52-card pack with no cave Priests. Because there is only one Earthquake and one of each artifact, several “two of the same” rules (double earthquake, etc.) cannot occur with the standard deck — but the code still handles them for variant decks.

Coordinate scheme. Areas live on an integer grid. A move N/E/S/W changes (x,y) by a unit vector; a stair U/D changes level by −1/+1 keeping (x,y). The Gateway starts at level 1, x=50, y=50 (packed 15050). North decreases y; South increases y; East increases x; West decreases x.

4. The Turn Lifecycle

↑ top

One iteration of the turn loop:

loop:
  if GS != PLAYING: break to scoring
  paint the turn screen (§12)
  k = read_key()                      # blocking single keystroke
  switch k:
    Q            -> confirm; if yes GS=QUIT
    X            -> if level==1 and stair-up present: confirm; if yes GS=ESCAPED
    M            -> open map browser; redraw; continue
    ?            -> open help; redraw; continue
    U            -> if level==1 and stair-up present: offer "leave cave" (== X);
                    else treat as move-up
    N/E/S/W/D/U  -> movement (arrows map to N/E/S/W)
    else         -> ignored
  if movement key:
    pool_pre_move(dir)                # Deep-Pool bookkeeping hook
    moved = try_move(dir)             # §6
    if moved:
      TN += 1
      pool_after_move()              # drop / reclaim heavy treasure
      resolve_area:                  # §4.1
        decode current card
        if special is DeepPool/ViperPit: enter_special()  (§10)
        elif chamber flag set:        enter_chamber()      (§7)
          if a trap dropped the party (FELL):
            relocate one level down (same x,y); goto resolve_area
    else if dead_end:
      show "that way is a dead end"; wait for a key
  # back to loop

4.1 Important sequencing facts

  • The turn counter advances only on a successful move. Failed moves (no exit, dead end, empty pack) do not consume a turn; neither does opening the map or help.
  • A trap fall is part of the same turn. Falling re-enters area resolution one level down and can chain (two traps stacked ⇒ two falls) without incrementing TN.
  • Death is detected lazily during the repaint: if a repaint finds zero living members, it sets GS=DEAD. Combat also sets GS=DEAD directly on a wipe.

5. Randomness & Shuffling

↑ top

A single linear-congruential generator (glibc constants) backs every random decision. Reproduce it exactly if you want identical sequences from a given seed.

seed ← (seed × 1103515245 + 12345) mod 2^31      # 31-bit; clear the sign bit each step

init_rng():       seed ← low 32 bits of system clock, forced ODD (set bit 0)   # avoid seed 0
next_rand():      advance seed; return it
roll_die():       advance; take bits 15..30 (a 16-bit value 0..65535);
                  v / 10923 (== 65536/6) clamped to 0..5; return v+1     # uniform 1..6
rand_below(n):    if n≤0 return 0; advance; return (bits 15..30) mod n
shuffle(arr,n):   Fisher-Yates: for i=n-1 down to 1: j=rand_below(i+1); swap arr[i],arr[j]
The generator extracts the upper bits (15..30), not the low bits, because LCG low bits have short periods. A browser port may substitute any decent PRNG; only a deterministic-replay/testing mode needs the exact constants above. Die rolls must be uniform on 1..6.

6. Map, Movement & Levels

↑ top

Exploration reveals the cave one card at a time. The party announces a direction; if the destination matches, the party moves; if it is a dead end, the card is placed face-down and the party stays put.

To explore an area, a player announces through which doorway his party intends to leave… If his chosen doorway leads to a dead end on the new card, he places the card face down and leaves his party where it is… But if his chosen doorway matches a doorway in the new area, the player leaves the card face up and moves his token onto it.

6.1 The move algorithm try_move(dir)

Directions: 1=N, 2=E, 3=S, 4=W, 5=Up, 6=Down. Returns moved? (boolean) and may set a dead_end flag.

try_move(dir):
  clear dead_end
  decode current card (PA)
  if the chosen direction has no exit/stair on this card: return false

  compute target coords from current ML[PA] using the unit vector:
     N: y-1   E: x+1   S: y+1   W: x-1   Up: level-1   Down: level+1

  search placed areas for one whose ML == target coords:
  ---- EXISTING area found ----
    if dir is a stair (Up/Down):
        # stairs always connect (see secret-door note)
        move there; PA←found; PL←new level; return true
    else:                            # lateral move needs a matching reverse doorway
        if destination has the opposite exit (N↔S, E↔W):
            move there; return true
        else:
            set dead_end; prune this direction's exit bit on the CURRENT card; return false
  ---- NO existing area ----
    if large pack exhausted (LI ≥ 60): return false
    draw next card index from LP[LI++]; value = AREA_CARDS[index]
    if target level == 1: clear the stair-up bit on the drawn card   # only the Gateway exits level 1
    if dir is a stair OR destination has the matching reverse doorway:
        place card at NM++ with target coords; if descending, mirror a stair-up onto it;
        PA←new slot; PL←new level; return true
    else:
        place card FACE-DOWN at NM++ (still recorded, reusable later);
        prune this direction's exit bit on the CURRENT card; set dead_end; return false
Dead-end pruning. When a direction proves a dead end, the implementation permanently clears that exit bit on the current card, so the player is never offered the same doomed direction twice. The placed face-down card stays in the map and may be entered later from a different direction (rules: “later a party may be able to enter this area from another direction”).
Secret doors / stairs. The rulebook says a stair often connects to an area with no matching stair pictured (one end hidden by a secret door). The implementation simplifies this for solitaire: stairs always connect in the direction travelled, and descending mirrors a stair-up onto the destination so you can climb back. Conditional secret-door knowledge across parties is irrelevant in solitaire and is not modelled.
Level-1 ceiling. Any stair-up drawn onto a level-1 card is suppressed, so the only way off level 1 (and out of the cave) is the Gateway's own stair-up. Descending always creates/returns the area directly below at the same (x,y).

6.2 Leaving the cave

Any stairway leading up from the first level is an exit from the Cave. Once a party leaves the Cave by one of these stairways, it may not return.

Pressing X (or U) when on level 1 with a stair-up present prompts a confirmation and, if accepted, sets GS=ESCAPED. In practice this means returning to the Gateway and climbing out.

6.3 Dead-end deadlock (forced redraw)

If all available doorways and stairways have been tried, the last area card played to make a dead end may… be put back into the middle of the pack, and another one drawn, until a way is found. This course cannot be followed when there is any other means of continuing the exploration.
The reference build does not implement the “return the dead-end card to the middle of the pack and redraw” rule. Because dead-end exits are pruned, a fully boxed-in tunnel can become a soft lock (no legal move). To implement: after a move fails, test whether every remaining exit/stair of the current card leads to a known dead end; if so, return the most-recently-placed face-down card to a random middle position in LP and allow the player to retry the same direction with a fresh draw. Guard it so it fires only when no other progress is possible.

7. Chambers, Draws & Hazards

↑ top
If the chamber is on the first level he draws one card; if on the second level two; third level three; fourth or deeper four… The Great Hall: draw 2 extra small cards. Tomb of Kings: draw 1 extra.

7.1 Entering a chamber

enter_chamber():
  if this area already visited (MV[PA]):       # revisit
     reload its persisted contents (AC) into the chamber working set (NS/NT/NH, CS/CT/CH)
     if empty: return silently
     re-fire any persisted hazards (Medusa/Ghouls re-roll; Trap re-checks;
        Earthquake & Mutiny are one-shot and never persist)
  else:                                          # first visit
     mark visited
     draw_count = min(level, 4)
     if special == Tomb:       draw_count += 1
     if special == Great Hall: draw_count += 2
     draw_count = min(draw_count, 8)
     draw draw_count cards from the small pack (stop early if pack empty);
       classify each: 100+ → stranger (max 8), 200+ → treasure (max 8), 300+ → hazard (max 4)
  apply_hazards()      # §7.2, in fixed priority order
  resolve_traps()      # may set FELL and end the turn here
  if party fell: return
  if strangers present: run the stranger encounter (§8)
  if no strangers remain (all defeated or joined): pick up treasure (§7.3)
  persist remaining contents back to AC; wait for a key

7.2 Hazard resolution

When two or more different hazard cards are drawn in the same chamber, they are dealt with in the order… Earthquake, Medusa, Ghouls, Mutiny, Trap… and before any other cards in the chamber are dealt with.

The implementation resolves in exactly that order. Each hazard:

HazardImplemented behaviour
EarthquakeCollapses the previous area PP: wipe its card, flags, and persisted contents (it becomes impassable). If two earthquake cards are drawn (variant decks), also collapse PP2 (two moves back). implemented
MedusaFor each living member roll 1d6; 1–2 turns that member to STONE (status 2) and drops their carried treasure into the chamber. A Wizard carrying the Magic Staff is immune (no roll). Strangers are unaffected. implemented
GhoulsEach living member fights ghouls of strength 2: member rolls 1d6 + fighting-strength vs 1d6 + 2; on a loss the member is killed (and drops treasure). No surprise either side. Talisman ward & real heavy-treasure-drop not implemented (flavour text only)
MutinyAll allies desert and join the chamber's stranger pool (re-testable later); their treasure is dropped. If the party is entirely allies, one ally (the first encountered) stays loyal. The party array is then compacted. implemented
TrapThe whole party falls one level (same x,y), ignoring this chamber's other cards, and resolves the area below. A living Dwarf negates a single trap; two traps in one chamber fall regardless. If a guiding Dwarf is later killed in a fight in this chamber, the party falls on leaving. implemented

7.3 Treasure pickup & carrying

If he has drawn unguarded treasure cards, he may pick them up and distribute them among members of his exploring party, taking care that no creature is given more heavy treasure than it will carry.

Carrying is bounded by weight only: a member may hold heavy treasure up to its kg capacity; weightless artifacts are effectively unlimited (a generous 20-slot backing array per member never binds in practice).

  • Treasure is offered only when no stranger remains in the chamber — every stranger must be defeated or have joined. If any stayed indifferent, the treasure stays put (rules: strangers guard treasure).
  • Pickup is a manual per-item assignment: the player picks a recipient from a roster cursor; the engine validates a free slot and the weight limit. If an item is too heavy for the chosen member, the player may drop carried items back to the chamber to make room, or leave the item.
There is no auto-distribution; assignment is always an explicit choice. Items left behind persist in the area's AC and can be collected on a later visit.
Treasure Chest opening, Lost Ruby statue, and Eye-of-God curse are not triggered during pickup. The Chest is treated as a plain 100 kg heavy item (only a Giant or Ogre can carry it); opening it for the die-roll loot table is not implemented. The Lost Ruby is picked up as ordinary points with no statue fight. Dropping the Eye of God imposes no curse. Each is specified individually in §11/§16.

8. Stranger Encounters

↑ top
When a party has entered a chamber containing strangers, it must in the same turn adopt one of the following three courses of action: withdraw… attack… or approach the strangers to test their reaction.

8.1 The three options

KeyOptionEffect
WWithdrawMove back to the previous area (PP). Strangers and their treasure stay. Turn ends. Always available.
AAttackBegin combat with surprise to the party (FS=+1). Always available.
TTestRoll the reaction (below). Unavailable once the area is flagged permanently indifferent.

8.2 Leader determination

Determine which of the strangers is the leader of the group according to priority… Spectre, Dragon, Wizard, Hero/Woman-Hero, Priest, Man/Woman, Giant, Ogre, Troll, Dwarf. The other strangers will react in the same way as the leader.

The implementation picks the stranger with the highest leader-priority value (column in §3.2; Sorcerer 11 is highest, Unicorn 0 lowest). Ties resolve to the first such stranger in draw order.

8.3 The reaction roll

roll = roll_die()                 # raw 1..6
natural1 = (roll == 1)
if any living party member has the CHARISMA flag (Hero/W-Hero): roll += 1
roll -= NC                        # curses
clamp roll to [1..6]
if natural1: roll = 1             # a natural 1 is ALWAYS 1, ignoring all bonuses

look up the LEADER's thresholds:
   roll ≤ hostile_max     -> HOSTILE
   roll ≤ indifferent_max -> INDIFFERENT
   else                    -> FRIENDLY
A bonus on this die roll (through having a hero or the ring) does not apply to a score of 1, which will always cause potentially unfriendly strangers to attack.
The Ring does NOT modify the reaction roll in this build — only Hero/W-Hero charisma (+1) and curses (−1 each) apply. The rulebook arguably grants the Ring's +1 here too; treat the charisma-and-curses-only behaviour as the normative implemented rule, and the Ring's reaction bonus as an optional extension.

8.4 Outcomes

OutcomeEffect
HostileCombat begins immediately with surprise to the strangers (FS=−1).
IndifferentIncrement this area's indifferent-count AI. On the 3rd indifferent result the area is flagged permanently indifferent — the strangers can never be tested again and forever guard their treasure. Turn ends.
FriendlyAll strangers join the party as allies (status 1), up to the party cap of 12; the chamber's strangers are cleared so its treasure becomes collectable. Turn ends.
Solitaire permanent-indifference realises the rulebook's solitaire rule exactly: three indifferent rolls ⇒ permanent. The count is per-area, persisted in AI[area], and the T option disappears once the perm-indiff flag is set.

8.5 Special creatures

  • Sorcerer — reaction thresholds make every roll hostile (always attacks). The rulebook's “may not approach to test” restriction is achieved by outcome (testing always yields hostile) rather than by removing the T option.
  • Unicorn — thresholds make it always friendly to everyone in this build (see gap below).
  • Dragon — always hostile by threshold.
Unicorn “friendly only to a party with a Woman” is not implemented. The canonical Unicorn is friendly if the party contains a Woman (or W-Hero, who has the befriends-unicorn flag) and otherwise stays indifferent, guarding treasure, and may only be approached after other strangers are dealt with. The build's Unicorn reaction table yields friendly for any roll. To implement: special-case creature 13 in the reaction logic — if a living party member carries the befriends-unicorn flag, friendly; else indifferent; and prevent it from being chosen leader until other strangers are resolved.
Charmed-Flute pre-emptive Dragon sleep is not implemented. The rules let a flute-bearer lull a Dragon to sleep before leader determination, so a friendlier creature leads. To implement: before find_leader(), if any living member carries the Charmed Flute and a Dragon is among the strangers, mark the Dragon asleep (exclude from leadership and from the fight) for as long as the party stays in the area.

9. Fights

↑ top
A fight may last one or more rounds, each round ending a turn of play… The player lays out the stranger cards in a line, and then pairs off his own fighting creatures against them… A die is rolled for each side, with that side's total strength… being added to the score. The side with the higher score wins that match, and the opponent is slain. If the score is tied, no one is slain.

9.1 Pairing — the focus-fire model

Combat is automatic with one player decision: the focus target. Each round:

  1. The player chooses which living stranger to focus (cursor left/right over strangers, confirm). Default focus is the strongest stranger.
  2. Casters (living Priests/Wizards) support the focus match, contributing magical power.
  3. The first non-caster front fighter anchors the focus; remaining non-casters each cover the next uncovered stranger, or pile onto the focus if every stranger is already covered (gang-up / focus-fire).
  4. Each stranger that has at least one front fighter is a primary match; strangers with no front fighter fold their strength into the focus match.

This realises the rulebook's “send two against one” (when the party outnumbers strangers) and “send one against two” / “fight the strongest” (when outnumbered), plus priests/wizards fighting from the background.

9.2 Surprise

The party or group of strangers which gains the advantage of surprise in a fight adds 1 to all its die rolls in the first round of the fight.

FS holds surprise: +1 party (set by Attack or by arriving fresh through a new doorway), −1 strangers (set when strangers turn hostile on a test), 0 none. The +1 applies to that side's strength in round 1 only.

9.3 Match resolution & strength bonuses

For each primary match, both sides compute total strength, each rolls 1d6, totals are compared. Higher wins (the strongest creature in the losing group is slain); a tie kills no one.

9.4 Special combat rules

In any round of a fight in which a party does not have any magical power to pit against a spectre, the strongest creature in the party must be matched against the spectre, and is automatically slain.
  • Spectre — if a Spectre is among the strangers and the party has no magical power and no Magic Sword bearer, the strongest living party member is slain each round before any dice. A party with any caster MP or a sword fights normally. implemented
  • Dragon-slayer — defeating a Dragon in a 1-on-1 match increments that fighter's PK; thereafter +PK is added to its strength (the dragon card is “inverted and put with its slayer”). implemented
Ring invincibility (level 4+) is not implemented. The canonical Ring makes its bearer immune to death on level 4 and deeper (death rolls ignored). The build applies only the +1 roll bonus. To implement: when a match would slay the Ring-bearer and level ≥ 4, ignore the death.

9.5 Heavy-treasure drop, retreat & casualties

Any members of the exploring party who are to fight hand-to-hand must drop any heavy treasure they are carrying… A party which retreats must leave behind any treasure dropped in the area.
  • Retreat vs. continue. After each round (if both sides survive) the player chooses Fight-on or Retreat. Retreat moves the party to the previous area and ends the encounter.
  • Casualty selection. When a match is lost, the weakest front fighter in that match dies. The rulebook's “player states which he prefers, then rolls 4–6 to get his preference” refinement is simplified to always-weakest.
  • Win / loss. All strangers dead ⇒ victory (return to exploration). All party members dead ⇒ GS=DEAD.
Heavy-treasure drop-before-combat and the blocked-retreat rule are not implemented. Fighters keep their heavy treasure through a fight, and a retreat into a dead end does not force another round. To implement: at fight start move every heavy item (ids 0,1,2,14) from fighters to the area, restore on victory, forfeit on retreat; and if a chosen retreat direction is a dead end or blocked by hostiles, force another round in the same turn.
Healing Balm is not implemented (no revive of a just-killed member). To implement: remember the last creature killed this turn; offer, at the start of a fight-free turn, a Woman/Priest/Wizard bearer to revive it, then consume the balm.

10. Special Areas: Viper Pit & Deep Pool

↑ top
On first turning the card over you move your token onto it, but not onto the narrow ledge [/across the water]. On the following turn you may either go back… or cross… and proceed through another doorway.

10.1 Viper Pit (special type 3)

On entry the party stops at the doorway; the turn ends. On the crossing turn:

  • For each living member, roll 1d6; a 1 means the member falls in and dies, their carried treasure lost to the pit (recoverable only with the Charmed Flute).
  • If any living member carries the Charmed Flute, all rolls are skipped — the vipers sleep and everyone crosses safely.
Simplification: the rulebook's per-segment crossing (1 segment for adjacent exits, 2 for opposite, one turn each) is collapsed into a single crossing event with one roll per member. Recovering lost pit-treasure with the flute is not modelled (the flute only prevents falls).

10.2 Deep Pool (special type 2)

If a giant has to carry across more than one load of heavy treasure there is a delay of one turn per additional load… Heavy treasure that has to be left behind is left in the doorway… recoverable only by a giant.

On entry the party stops at the edge. When the party next moves out through a different doorway than it came in:

  • Each non-Giant member drops its heavy treasure (ids 0,1,2) into the doorway it exited from (recorded in DROP_SIDE / DROP_PT, up to 4 items). Artifacts (weightless) are kept.
  • A living Giant carries all heavy treasure across.
  • Re-entering the pool and leaving by the same doorway where treasure was dropped offers to reclaim it (returned to the first member with room).
Simplification: a single living Giant carries all heavy loads at once, with no per-load turn delay; the multi-load Giant delay is not modelled. Up to 4 dropped items are tracked per doorway.

10.3 Tomb of Kings & Great Hall

These are ordinary chambers handled entirely by the extra-draw rule in §7.1 (+1 and +2 cards respectively). There is no special crossing behaviour.

12. Scoring & Game Over

↑ top
Players whose parties succeed in leaving the Cave receive points for all cards they hold… Dragon-slayers have double value. A party which has killed the Sorcerer receives a bonus of 30 points. A party under a curse deducts 30 points.

12.1 The scoring formula

score = 0
for each party member with status ORIGINAL(0) or ALLY(1):       # not stone, not dead
    pts = creature_points[PC]
    if PK > 0: pts *= 2                  # dragon-slayer doubles the creature's points
    score += pts
    for each treasure carried by the member:
        score += treasure_points[treasure]
if sorcerer killed (SK):  score += 30
score -= 30 * NC                          # 30 per curse (cumulative)
score = max(0, score)

if GS == DEAD:  score = 0                  # a wiped party scores nothing
Curse penalty scales: −30 per curse, not a flat −30. The dragon-slayer doubling applies to the creature's own points only, not its treasure.

13. Display & Interaction (Logical UX)

↑ top

The screen is a single rectangular canvas with a border. The game paints text and line-art into logical regions of that canvas and repaints the whole thing each turn inside one batched update. A browser port should treat each region as an independent component.

13.1 The turn screen — logical regions

RegionPurpose
Status / headerLevel, current area name (Tunnel / Chamber / special name), turn number, count of areas explored.
Area visualizationLine-art of the current area with content glyphs (§13.2). Replaces a textual exits list — exits and stairs are shown visually.
Party rosterOne line per living member: name + carried items, framed in a titled box. Stoned/dead members are omitted; an all-empty roster triggers game-over.
Prompt / messageCentered action prompt or the latest event sentence (hazard banner, encounter result, etc.).
The original design envisaged seven independent panels (status, party, map, chamber, legend, log, prompt). The shipped build consolidates these into one canvas with painted regions plus separate full-screen modes for the map browser and help. A re-implementation is free to use either structure; the region semantics above are what matter.

13.2 Area visualization vocabulary

The current area is drawn as line-art. Tunnels are a small crossroads with corridor arms only where exits exist; chambers are a larger rectangle with width-3 doorways and short exit stubs; special areas are a labelled rectangle (GATEWAY / POOL / VIPER). Content is placed by class:

  • Creatures (strangers): top-left, single-letter mnemonics.
  • Stairs: top-right, letters U / D.
  • Party: centre, single-letter mnemonics of living members.
  • Treasure: bottom-left, single-letter mnemonics.
  • Dropped heavy treasure: a $ in the doorway it was left in (Deep Pool).
     Chamber, exits N/E/S/W + stair-up, two strangers + treasure
     +----------------+
     | OT          U  |     OT = Ogre, Troll (top-left)   U = stair up (top-right)
     |                |
    -      H  P      -      H, P = party (centre)         - = E/W doorways
     |                |
     | $ G            |     $ G = Silver, Gold (bottom-left)
     +----------------+
            |  |              N/S doorway stubs

The glyph vocabulary (creatures & treasures) is in Appendix C. Letters are mostly the initial of the name, with case/letter disambiguation where initials collide. A browser port may instead use icons or colour-coded tiles, but should preserve the spatial grouping (who/where) so the player can read an area at a glance.

13.3 The map browser (full screen)

Pressing M opens a scrollable orthogonal map of the whole cave, one level at a time.

  • Each placed area is one glyph at its (x,y) cell. Current area is shown in reverse video.
  • Area glyphs: G Gateway, P Deep Pool, V Viper Pit, T Tomb, H Hall, O ordinary chamber, + tunnel.
  • Horizontal/vertical connectors (–, |) join areas with matching exits; stairs show ^ (one cell above) and v (one cell below), pointing to the same (x,y) on the adjacent level.
  • U/D change level (clamped 1..max); arrows scroll the viewport; M/Q/Enter return.
  • Header shows “Level n of m”; a legend sits at the bottom. The viewport recenters on the party on open.
Map cells are spaced (the reference build uses 4×2 character cells) so connectors are visible. The map shows structure only — no creatures, treasure, or hazards — per the original requirement.

13.4 Input model

Strictly single-keystroke, blocking, no echo of invalid keys. Universal keys during a turn:

N E S W — move (arrows alias)
U D — stairs up/down
M — open map
? — help
X — exit cave (level-1 stair-up)
Q — quit (confirm)

Modal keys: encounter W/A/T; fight F/R and focus-picker arrows + Enter; yes/no Y/N; treasure pickup roster cursor + Enter/Space/L(leave)/D(drop).

Inventory-redistribution (I) and an artifact menu (A) are not wired into the turn loop. The rules permit redistributing treasure among members at the start/end of a turn (when not fighting); the build only assigns treasure at pickup time. To implement: add a modal that lets the player move items between members subject to weight limits, and an artifact menu to invoke the active artifacts of §16.

16. Implementation Sketches for the Open Gaps

↑ top

Concrete, minimal designs for the “not implemented” rows, so a re-implementation can reach full fidelity.

Talisman

Passive. Before the Ghouls handler, if any living member carries the Talisman, skip the ghoul fight (“the talisman wards them off”). In a stranger encounter, if the strangers include a Spectre, the party holds the Talisman, and level ≥ 4, remove the Spectre from the encounter.

Lotus Dust

Active, one use. Offer before approaching strangers or before any fight round. Player targets one enemy; mark it asleep (excluded from leadership/matching) for 2 of the player's turns, then it wakes. Invalid on Spectres/Ghouls; works on Medusa (used before the Medusa roll skips it). Consume on use; sleeping creatures are protected by a curse if killed while asleep.

Magic Carpet

Active, one use, requires a living Priest or Wizard. Player picks a direction (N/E/S/W/U/D); compute target coords; place or reuse the area there (cannot be the cave exit). The party arrives with no withdraw option if it meets strangers; arriving at a Pool/Viper lands on the island. Grants surprise (arrival by carpet counts as a fresh entry). Consume.

Healing Balm

Active, one use, requires a living Woman/Priest/Wizard, only at the start of a fight-free turn. Track the last-killed creature this turn (id + prior status + treasure); revive it (restore status), re-insert into the party, consume the balm.

Lost Ruby

On attempting to pick up the Lost Ruby, trigger a one-match fight against a statue of strength 8 (one party fighter vs 8 + die). Win ⇒ ruby to inventory. Lose ⇒ fighter slain, ruby stays, set the area's statue-aroused flag (AF bit 0x20). On any later entry to an aroused-statue area, the statue attacks first.

Eye of God

While the Eye is in the party's area: priests/wizards and all other artifacts lose their magical effects (but keep fighting strength); Spectres are annihilated; the Lost-Ruby statue cannot attack. Once taken up, the Eye must remain with the party or a curse applies until it is retaken (including if left on a slain carrier). Picking it up is optional.

Treasure Chest open

An action available at the start of any in-cave turn while a member carries the Chest. Roll 1d6 and apply the table in §11. On a 2, run a one-round fight against a Spectre (MP 5); if undefeated it remains in the area, hostile. Two members may jointly carry the 100 kg chest (relax the single-carrier weight check).

Magic Staff reanimation

Active, requires a Wizard bearing the Staff. Restore any party member with STONE status (2) to alive.
