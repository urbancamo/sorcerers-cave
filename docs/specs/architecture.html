1. System overview

The Sorcerer's Cave is a browser game with a server-authoritative backend. Three pieces, in one pnpm monorepo, share a single rules engine written once in TypeScript. 

sorcerers-cave/                          — pnpm workspace
├─ packages/engine  (@sorcerers-cave/engine)   Pure, deterministic game rules
├─ packages/assets  (@sorcerers-cave/assets)   Card/tile manifest types
└─ apps/web                                     The deployable app:
   ├─ src/      React + Vite frontend (the browser client)
   └─ convex/   Convex backend (serverless functions + reactive DB)
LayerTechnologyResponsibility
EngineTypeScript (pure functions)Every game rule. No I/O, no randomness source but a seeded LCG. Runs on both server and client.
BackendConvex (functions + reactive document store)System of record & rules authority; auth, persistence, multiplayer coordination, live subscriptions.
FrontendReact + Vite, Three.js (vanilla)Rendering, input, and an optimistic local mirror of the engine for instant feedback.

The defining idea: the engine is the single source of truth for rules, and it is shared by both tiers. The backend imports it to resolve every action authoritatively; the frontend imports the very same code to predict the outcome locally so the 3D view responds instantly. Because the engine is pure and deterministic (a seeded random-number generator, no wall-clock), the two always agree.

2. The shared engine

packages/engine is a dependency-free TypeScript library. It exposes an immutable-ish reducer at its heart: 

reduce(state: GameState, action: GameAction): { state: GameState; events: GameEvent[] }
  • GameState (state.ts) — the entire game as a plain, JSON-serializable object: the placed areas, both card decks + their draw indices, the party, the live encounter/fight, the RNG seed, score, curses, etc.
  • GameAction (actions.ts) — every player decision: move, test, attack, fightOn, retreat, takeTreasure, useArtifact, quit…
  • GameEvent — what happened (combatRoll, memberDied, drewChamber, eyeForsaken…). The reducer is the only producer; the UI never infers game facts, it just renders events.
  • legalActions(state) (selectors.ts) — the set of currently valid actions, so the UI offers exactly what the rules allow.

Rule modules behind the reducer: map/coords/decode (the tile grid), chamber/decks (drawing), reaction/combat (encounters & fights), hazards, special (Deep Pool / Viper Pit), pickup, effects (artefacts), score, and rng (the seeded LCG that makes everything reproducible).

Multiplayer core — compose / decompose

multi.ts adds multiplayer without forking the rules. One game holds a single shared CaveState (the map + both decks + the RNG stream) plus an array of per-seat PartyState. To apply a seat's action it composes a single-party GameState (cave ⊕ that party), runs the same reduce(), then splits the result back. The engine therefore remains the sole rules authority, and anything stored on a party (curses, permanent indifference, retreat-hostility) is naturally per-party. 

mpReduce(mp: MpGameState, seat, action): { state: MpGameState; events }
   // turn-gated: rejects an action from a seat whose turn it isn't
Why it matters One implementation of the rules, exercised by hundreds of unit tests, runs identically server-side (authority) and client-side (prediction). No rule logic is duplicated in React or in Convex.

3. Backend architecture

The backend is Convex: a reactive document database with serverless queries (read, cached, live) and mutations (transactional writes). Functions live in apps/web/convex/ and are written in TypeScript against a typed schema; npx convex codegen generates a fully-typed client API (convex/_generated/api).

The authority pattern

Every mutation that changes a game follows the same shape — load, run the shared engine, persist:

// solo: game.applyAction
const game = await ctx.db.get(id);
const { state, events } = reduce(game.state, action);   // the shared engine
await ctx.db.patch(id, { state });
return { events };

// multi: multiplayer.act — turn-gated, then broadcast
const { state, events } = mpReduce(mp, me.seat, action);
await ctx.db.patch(gameId, { state });
// derive + post narration/finish lines to `messages`; record finished parties to highScores

The client's request only ever names an action; the server decides the outcome. Read access is membership-gated — multiplayer queries return null to non-members, and project the state per seat so a player sees the shared cave plus their own party.

4. Frontend architecture

A Vite + React single-page app (apps/web/src). It splits cleanly into a view layer (rendering the cave) and a game-UI layer (panels, dialogs, flows), bound together by a small set of hooks and a single rendering port.

Composition root & navigation

main.tsx → ConvexProvider(client) → App.tsx → GameScreen.tsx
   ├─ SplashScreen           title, resume-by-code, multiplayer entry, link to /manual.html
   ├─ (solo) CaveCanvas + panels
   └─ (multi) MultiplayerSetup / MultiplayerLobby / MultiplayerGame

The rendering seam — ports.ts

The 3D renderer is deliberately decoupled from the engine by an interface, CaveEngine (view/ports.ts). It exposes a render-friendly view (placed Areas, the current tile, openMoves(), state()) and the few imperative calls the renderer makes (tryMove, exit, quit). The golden rule, stated in the file: the engine owns every rule; the renderer only draws and forwards intent.

  • Feedback — DiceRoll, NoticeModal, and the pure event-to-UI mappers rollView.ts (events → a dice overlay) and eventNotices.ts (events → notices).

5. Frontend ⇄ backend communication

There is no bespoke API or socket code. The frontend reads through Convex's reactive queries (live subscriptions) and writes through mutations — and keeps a local engine mirror so the 3D view never waits on the network.

Reactive reads, transactional writes

  • Subscriptions. Components call useQuery(api.…). Convex pushes a fresh result over a websocket whenever any data the query touched changes — so every client re-renders automatically. In multiplayer, one player's act updates the shared games row and all seats' playView/gameState subscriptions update.
  • Mutations. Actions are sent with useMutation(api.…). The server runs the engine and returns the resulting events (used to drive dice/notice overlays).

Server authority + optimistic mirror

The renderer needs an instant answer when you click a doorway, but the server is the authority. The adapter resolves this by running the same engine locally first, then reconciling: 

click doorway ──▶ engineAdapter.tryMove(dir)
                   │  local reduce()  ──▶ scene updates immediately (optimistic)
                   └─ onAction ──▶ useMutation act/applyAction ─▶ server reduce() (authority)
                                                                          │ persists state
reactive useQuery ◀── new authoritative snapshot ◀──────────────────┘
   └─ adapter.sync(state)   reconcile the local mirror to the server's truth

Because the engine is deterministic and the RNG seed travels inside the state, the optimistic result and the authoritative result match — reconciliation is normally a no-op. Move-triggered dice (e.g. ghouls fighting on entry) are surfaced from the mutation's returned events.

Multiplayer specifics

  • Turn-gating. act rejects an action from a seat whose turn it isn't (the engine's mpReduce enforces it); the UI only enables controls on your turn.
  • Per-seat projection. playView returns the composed state for your seat plus every party's position/colour; spectateView returns any seat's view so you can follow along; gameState feeds the live scoreboard. All are membership-gated.
  • The feed. Chat and auto-narrated action broadcasts (defeats, pickups, descents, finishes) are rows in messages; clients subscribe and render them as the chat log and as transient toasts.
  • Identity. Players are authenticated via Convex Auth; a four-letter game code is the shareable handle to join or resume.

6. Build & deployment

  • Monorepo — pnpm workspaces; the engine/assets packages are consumed from TypeScript source, so a single typecheck spans the whole repo.
  • Frontend — Vite builds apps/web to static assets; the Convex deployment URL is injected at build time as VITE_CONVEX_URL, and feature flags (e.g. VITE_MULTIPLAYER) gate optional surfaces.
  • Backend — npx convex deploy bundles the functions (and the imported engine) and pushes the schema to the Convex deployment. On Vercel this runs inside the build command, so a single deploy ships both tiers in lock-step.
  • Tests — Vitest: the engine carries the bulk (rules), with Convex function tests (convex-test) and React component tests (Testing Library) in apps/web.
