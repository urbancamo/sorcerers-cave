1 Goals & scope

Let up to four people, each in their own browser, explore one shared Cave as separate parties, taking turns according to the board rules — joining via the existing four-letter game code.

  • One shared cave, many parties. The map, the area (large) deck and the small-card deck are shared; each player commands their own party with its own position, members, treasure and fortunes.
  • Join by code. The four-letter code already minted for save/resume becomes the invitation: share it out-of-band, others enter it in their own browser to take a seat.
  • Host-controlled lobby. Up to four seats. The creator starts the game when ready; once started, no one else may join.
  • Named parties. Each player must give their party a required Party Name when creating or joining — it identifies them everywhere (lobby, token labels, turn banner, chat, results).
  • Unique colours. Each seat picks an unused party colour; only free colours are offered.
  • In-game chat. A simple broadcast message feed lets players talk to each other (simulating table-talk) — an MVP feature alongside the lobby.
  • Everything is turn-based — including party selection, because every party is drawn from the same finite small pack.
  • Faithful to the board game first; advanced inter-party interactions (fights, unions, trading) are staged and optional, matching the rulebook's own "beginners vs. serious play" split.
Guiding principle The engine stays the single source of truth for rules; Convex stays the single source of truth for authority and turn order; Convex reactivity is the real-time sync layer (no extra websockets needed). This mirrors the existing solitaire architecture.

2 How multiplayer works, per the rules

Interpreted from the rulebook (docs/specs/sorcerers-cave-rules.md), with the exact online mapping.

2.1 Turn order & party selection

  • Order of play is decided by a die roll. The rule: "the player who has first choice of an exploring party is the last to move, and vice versa." → We roll a random play order; party selection runs in the reverse order, so first-pick = last-to-move (compensating the pick-first advantage).
  • Party selection is turn-based from the one small pack. On a player's pick turn they choose creatures (selection value ≤ 6) from the cards still in the pack; those cards are removed so no one else can draw or pick them. After all seats have picked, the remaining pack is the shared chamber draw pile.
  • This is already half-built: solitaire newGame removes a single party's picks from the shared pack. Multiplayer generalises it to sequential picks during a dedicated partySelect phase.

2.2 A turn (exploration)

"It takes one turn to explore one area." On their turn a player either moves one area into already-explored space, or announces a doorway and draws the top large-pack card:

  • Dead end → card laid face-down, party stays (turn ends).
  • Matching doorway → card face-up, token moves on. Tunnel / pool / pit → turn ends. Chamber → draw small cards (1–4 by level) and resolve hazards / treasure / strangers — exactly today's solitaire flow, but acting on that player's party.

2.3 The cave is shared; chambers persist

Key insight — already supported Leftover strangers, treasure and hazards are persisted onto the area (PlacedArea.contents) and reloaded when any party (re)enters. So a chamber a previous player left half-fought is seen by the next party that walks in — "other parties entering the chamber have the usual options." The shared map + per-area contents model already carries this; we only need each party to carry its own transient working set during its own turn.
  • The first party to enter a fresh chamber draws the small cards; later parties draw none, but must deal with whatever remains.
  • Strangers that turned hostile to a party and were retreated-from stay hostile to that party but may be approached afresh by others — a per-party relationship flag (advanced; see gaps).
  • Secret doors are per-discoverer: only the colour(s) who explored the stairway (or were shown it) may use it. Today secretDoors/PlacedArea.secretDoor is global; multiplayer needs per-player knowledge.

2.4 Leaving & scoring

Any up-stair on level 1 is an exit; once a party leaves it cannot return. "Play ends when all parties able to do so have left the Cave." Each party scores independently (creatures + treasure, dragon-slayers doubled, Sorcerer bonus −30-curse), and the highest score wins. A Sorcerer bonus is split if parties combined against him.

2.5 Inter-party interaction (advanced / optional)

The rulebook explicitly recommends that beginners disallow party-vs-party interaction. We adopt that for the MVP and stage the rest:

Shared visibilitySame-area parties see each other; "hidden cards" is a serious-play variationMVP (all face-up)

3 Gaps in the rules & the code

3.1 Functionality gaps (today's code is single-party)

Biggest item The engine's GameState models exactly one party — a single party, partyArea, working set (strangers/treasures/hazards/sleeping/lulled), fight, curses, turn, seed. Multiplayer needs one shared cave + N per-player parties. This is the core refactor (see §4–5).
  • Server is owner-scoped & single-player. games holds one state and one color; applyAction/get enforce ownerId === caller (an IDOR guard). Multiplayer needs multiple authorised players per game and turn-gating instead of owner-gating.
  • No lobby / seats / colour reservation / start-lock exist yet.
  • Join by code is single-player only. resumeByCode restores one's own game; multiplayer needs a joinByCode that adds a seat to someone else's game (without claiming ownership).
  • Turn-based party selection doesn't exist (solitaire picks once, locally, then creates the game).
  • Per-player secret-door knowledge and per-party stranger-hostility memory aren't modelled (currently global / absent).

3.2 Rules gaps (the board game assumes humans in a room)

  • Disconnection / abandonment. The rules have no notion of a player dropping out. We must decide turn-timeouts, host "skip seat / remove seat", and what a removed party's cards do (left in place? removed?).
  • Simultaneity & the defender's rolls. "It is always a player's privilege to roll the die for his own scores" and inter-party fights alternate across turns — i.e. some actions belong to a non-current player. The MVP (no party fights) sidesteps this; Phase 5 needs an explicit "reaction window" for the defender.
  • Hidden information. Competitive play hides most cards; the engine/transport would need per-player views. We default to the rulebook's beginner recommendation (all cards shown) and treat fog-of-war as a later variation.
  • Physical "show the cards" / arranging the map have no digital analogue — handled implicitly by shared reactive state.

4 Target architecture

Two clean layers, same as solitaire, generalised:

Engine pure TS
A multi-party state = one Cave (shared map + decks + RNG) and an array of Parties (one per seat) + turn order. A new reducer mpReduce(game, seat, action) → {game, events} validates that seat may act, then dispatches to the same primitives solitaire already uses (move, enterChamber, combat, pickup) operating on that seat's party against the shared cave.
Convex authority
Owns identity, the seat roster, colour reservation, turn order and the start-lock. applyAction becomes turn-gated (the caller must occupy the current seat). Reactive queries push every change to all browsers — no extra realtime infra.

4.1 Reuse, don't rewrite

Most solitaire reduce logic already operates on "the party + the cave." The refactor extracts a party context so the same code runs per-seat:

  • Shared on the cave: areas, largePack/largeIdx, smallPack/smallIdx, seed.
  • Per party: members, partyArea/prev/prev2, the transient working set, fight, curses, bonusScore, sorcererKilled, surpriseReady/fellThroughTrap, secret-door knowledge, status.
Migration tactic Keep solitaire working throughout. Introduce the multi-party types alongside the existing GameState; implement mpReduce by delegating to refactored, party-context primitives. Solitaire can later be re-expressed as "a one-seat multiplayer game," but that convergence is optional and comes last.

5.2 Engine state shape

interface CaveState {            // shared by all parties
  areas: PlacedArea[];          // the one map (per-area contents already shared)
  largePack: number[]; largeIdx: number;   // area deck
  smallPack: number[]; smallIdx: number;   // small deck (party picks removed in turn)
  seed: number;                 // single authoritative RNG stream
}

interface PartyState {          // one per seat
  seat: number; color: PartyColor; name: string;   // name = the required Party Name
  members: PartyMember[];
  partyArea: number; prev: number; prev2: number;
  strangers: number[]; treasures: number[]; hazards: number[];
  sleeping?: number[]; lulled?: number[];   // current chamber working set
  fight: FightState | null;
  curses: number; bonusScore: number; sorcererKilled: boolean;
  surpriseReady?: boolean; fellThroughTrap?: boolean;
  knownSecretDoors: number[];   // area coords this seat may use
  status: "selecting" | "exploring" | "left" | "wiped" | "abandoned"; // last three are terminal
  score?: number;               // frozen when the party reaches a terminal state
}

interface MpGameState {
  phase: "partySelect" | "playing" | "finished";
  cave: CaveState;
  parties: PartyState[];
  order: number[];              // seats in PLAY order (random)
  pickOrder: number[];         // = order reversed (first pick = last to move)
  active: number;              // index into the relevant order whose turn it is
  turnCount: number;
}

The reducer signature becomes mpReduce(g: MpGameState, seat: number, a: GameAction) → {state, events}; it rejects actions from a seat that isn't currently to move (except a future "defender reaction" in Phase 5).

6 Server functions (Convex)

FunctionKindBehaviour & guards
createMultiplayer({partyName,color})mutationrequire a non-empty partyName (1–24 chars); mint a unique code; insert games (mode:"multi", lobby:"open", hostId); insert the host's players seat 0 with party name + colour; post a "<partyName> created the game" system message. Returns { gameId, code }.
lobby({code})querypublic; returns seats (party names, colours, ready), host, lobby status, taken colours and taken names. Drives the join screen for everyone, reactively.
joinByCode({code,partyName,color})mutationonly while lobby==="open" & seats < max & colour free & partyName non-empty and unique-in-game & caller not already seated; assigns next seat; posts a "<partyName> joined" system message. Does not claim ownership. Idempotent if already seated (lets a player rejoin).
setPartyName / setColor / setReady / leaveSeatmutationlobby housekeeping; name change rejected if blank or taken, colour change rejected if taken; host leaving either promotes the next seat or closes the lobby.
startGame({gameId})mutationhost only; flips lobby:"started" (locks joins), rolls the random play order (seeded), builds the shared MpGameState in phase:"partySelect", sets turnSeat to the first picker.
pickParty({gameId,picks})mutationturn-gated to the current picker; validates value ≤ 6 & that each card is still in the pack; removes them; advances pick turn; when the last seat picks, transitions to phase:"playing" and sets the first mover.
applyAction({gameId,action})mutationgeneralised: resolve caller→seat; require seat === state.turnSeat (turn-gating replaces owner-gating); run mpReduce; persist state + turnSeat; append gameEvents with seat. End-of-turn advances to the next seat still in play.
get({gameId})queryreadable by any seated player (membership check via players), not just an owner. Returns the full reactive state for the renderer.
sendMessage({gameId,text})mutationmembership-gated (caller must be a seated player); trims & caps text at 280 chars, rejects empty; stamps it with the caller's seat/party name/colour and inserts a messages row. Allowed any time (lobby and in-game), on any seat's turn.
messages({gameId,limit?})querymembership-gated; returns the most recent chat + system lines (default ~100) oldest-first, reactively, for the chat panel.
results({gameId})queryonce phase==="finished", returns every party ranked by score (party name, colour, outcome, score, and the per-party detail) for the combined scoreboard. Public by code so players can compare afterwards. (See §8.4.)
Authority model Replace the single ownerId === caller check with: membership (a players row exists for caller+game) for reads, and turn ownership (caller's seat === turnSeat) for actions. The engine still re-validates every move, so the server stays the sole authority and the client is never trusted.

7 Client & UX flow

  1. Splash → "Start Multiplayer Game". Host enters a required Party Name and picks a colour, then gets a code shown prominently (reuse the save-modal styling) to share out-of-band. Start is blocked until a party name is given.
  2. Splash → "Join Multiplayer Game". Anyone enters the code (reusing the resume-by-code input), a required Party Name (must be unused in that game), and picks from the remaining colours.
  3. Lobby (reactive for all): list of seats by party name with colour chips and ready state; the host sees a Start button (enabled with ≥ 2 seats). Late-comers past started are told the game has begun. A chat panel is present from here on so players can coordinate.
  4. Turn-based party selection. A shared "draft" screen shows the small pack's remaining selectable creatures; only the player whose pick turn it is can choose (others see "Waiting for <party name>…"). Picks deplete the shared pool live.
  5. Play. The existing 3D cave renders the shared map with every party's coloured token, labelled by party name. A persistent banner shows whose turn it is (by party name); controls are live only on your turn (others watch the map update reactively). The HUD's Deck indicator (already shared) reflects the common large pack. The chat panel stays docked in a HUD corner with an unread badge.
  6. Chat. A docked, collapsible message feed: coloured bubbles tagged with each sender's party name, plus system lines (joins, turn changes, exits). Send on Enter; reactive for everyone. MVP scope = plain broadcast to all players in the game (no private/whisper messages).
  7. Leaving & results. Parties exit independently; when all have left/wiped, a multi-party results screen ranks the parties by party name and writes each to the leaderboard.

8 Finishing the game

Parties finish independently; the game finishes when none can carry on. An exited player isn't kicked out — they can keep watching, and everyone gets a shared scoreboard at the end.

8.1 When a single party finishes

A party reaches a terminal state in one of three ways:

  • Escaped — climbed out via a level-1 up-stair (cannot return).
  • Wiped — every member dead/stone.
  • Abandoned — the player voluntarily ends the expedition (the per-party equivalent of today's quit), for when "no one cares to go further."

On reaching a terminal state the engine freezes that party's score and sets its status; turn-rotation then skips that seat for the rest of the game. The player is shown their personal score screen (the existing solo GameOverScreen, scoped to their party).

8.2 Read-only spectating your suggestion

Exited ≠ gone From their score screen an exited player chooses Return to menu or Watch the cave. "Watch" drops them into the live cave in read-only mode: the shared map and every party's token update reactively, but no controls are offered (their seat is terminal, so turn-gating never gives them a move). Chat stays open so they can cheer or heckle.
  • This needs almost nothing new: get is membership-gated (an exited player is still a member), the renderer already reflects reactive state, and turn-gating already excludes terminal seats. The client just hides action controls when mySeat is terminal or it's not your turn.
  • An optional players.spectating flag (or reusing lastSeen presence) lets the UI show "👁 2 watching". A player who returned to the menu can rejoin by code at any time to watch or see results.

8.3 When the whole game finishes

The game flips to phase:"finished" when every seat is terminal — i.e. all parties have escaped, been wiped, or abandoned. It also ends if the large pack is exhausted and no still-active party can (or wishes to) move, in which case remaining parties are settled as abandoned where they stand. After each action the server checks this end condition and, on transition, records the combined results.

8.4 Combined results view your suggestion

One shared scoreboard Once finished, all participants (players and spectators) see a single combined results screen: every party ranked by score, with colour, party name, outcome (escaped / wiped / abandoned), final score, and an expandable per-party breakdown — the party & artifacts that left the cave and the expedition stats — reusing the high-score detail components we already built.
  • It's reachable by anyone in the game (and later via the game code), so players can compare at leisure.
  • Each party's result is also posted to the global leaderboard, tagged as multiplayer and labelled with the party name, so multiplayer runs sit alongside solo records (filterable).
  • A new query results({gameId}) returns the ranked per-party results once phase==="finished"; a finishParty/end-of-game step writes them. Chat remains visible beside the scoreboard.

9 Phased roadmap

Phase 1 — Infrastructure, lobby, names & chat no engine change MVP
Schema (games lobby fields + players + messages tables); createMultiplayer, joinByCode (both requiring a Party Name), lobby, setPartyName/setColor/setReady/leaveSeat, startGame (lobby flip only), and sendMessage/messages for chat. Splash buttons, join input, reactive lobby with party names + colour reservation + host start, and the docked chat panel. Deliverable: create a named-party game, share the code, join from another browser, see each other by party name, chat, and host can lock the lobby.
Phase 2 — Multi-party engine core
Introduce MpGameState/PartyState/CaveState; refactor solitaire primitives to a party context; mpReduce with turn-gating; build state on startGame with random play order. Unit-tested in isolation (no UI yet).
Phase 3 — Turn-based party selection
pickParty draft from the shared pack in reverse play order; transition to playing. Draft UI.
Phase 4 — Shared exploration, turns & finish
Turn-gated applyAction; render all parties' tokens; turn banner; end-of-turn advance; independent terminal states (escaped / wiped / abandoned) with per-party score; read-only spectating for exited players; end-of-game detection; the combined results screen (results query) + per-party leaderboard posts (see §8). Deliverable: a fully playable competitive game under the beginner ruleset (no party-vs-party), from lobby to shared scoreboard.
Phase 5 — Inter-party interaction advanced
Trading; party-vs-party fights with the cross-turn defender reaction window & surprise rule; per-party stranger-hostility memory; per-player secret-door knowledge.
Phase 6 — Unions, division & variations optional
Unions/forfeit-a-turn, party division/guarding, and the hidden-cards (fog-of-war) competitive variation via per-seat views.

10 Step-by-step: building a fully multiplayer game

The concrete sequence, from infrastructure to a finished competitive game.

  1. Schema. Add lobby fields to games; create the players table (with partyName) and the messages table, both with by_game indexes (players also by_user); add optional seat to gameEvents. Deploy (additive, back-compatible).
  2. Create + code. Implement createMultiplayer (require Party Name, mint code, host seat, lobby open) and surface "Start Multiplayer Game" on the splash with a required party-name field, showing the code to share.
  3. Join in another browser. Implement lobby (public, reactive) and joinByCode (require a unique Party Name, seat assignment, colour reservation, no ownership claim). Wire "Join Multiplayer Game" with the code + party-name field + free-colour picker.
  4. Lobby & colours. Render the reactive seat list by party name with colour chips; reserve colours (offer only unused ones); host sees Start.
  5. Chat. Implement sendMessage/messages and a docked, reactive chat panel (coloured bubbles tagged by party name + system lines), available from the lobby onward. MVP
  6. Start & lock. startGame: host-only, flip lobby:"started" (joins refused), roll random play order, build MpGameState in partySelect.
  7. Engine core. Land MpGameState + mpReduce + party-context refactor with unit tests (movement, chamber, combat, pickup per seat against the shared cave).
  8. Turn-based party draft. pickParty from the shared pack in reverse play order; draft UI showing the depleting pool and "waiting for <colour>"; transition to playing.
  9. Turn-gated play. Generalise applyAction to membership + turn checks; render every party token on the shared map; add the whose-turn banner; advance the seat at end of turn; skip seats that have left/wiped.
  10. Independent exits & finish. Each party reaches a terminal state (escaped via a level-1 up-stair / wiped / abandoned) and sees its score with a Watch the cave (read-only spectate) or Return to menu choice; turn-rotation skips terminal seats. When all seats are terminal, flip to finished, show the combined results scoreboard (ranked, expandable per-party detail) via results({gameId}), and post each party to the leaderboard. (See §8.)
  11. Resilience. Presence (lastSeen), a turn timeout / host "skip seat", and rejoin-by-code into a started game (membership already exists).
  12. Advanced interactions. Layer in Phase 5–6 (trading, party fights, unions, division, fog-of-war) behind a game-variant flag.

11 Open decisions & risks

  • Minimum players to start. Recommend ≥ 2 (a one-seat "multi" game is just solitaire). Host decides exact count up to 4.
  • Disconnection & stalls. No board-rule precedent. Proposed: a soft turn timer with a host "skip turn", and a host "remove seat" that leaves the abandoned party's cards in place (so others may still encounter/loot them). Confirm desired behaviour.
  • MVP interaction scope. Ship Phase 1–4 (beginner ruleset: no party-vs-party, all cards visible). This is a complete, fun competitive race and de-risks the hardest rules (cross-turn defender rounds, fog-of-war).
  • Engine refactor blast radius. The single-→multi-party change touches the core reducer; mitigate by keeping solitaire on its current path and building multiplayer beside it, converging only if/when it's clearly safe.
  • RNG. One authoritative seeded stream on the cave keeps every browser deterministic and replayable, as today.
  • Leaderboard. Decide whether multiplayer results post per-party (recommended) and whether they're tagged as multiplayer to keep solo records comparable. Party names make natural leaderboard labels.
  • Party names. Required, 1–24 chars, unique within a game. Open question: profanity filtering and whether to allow editing a name after the game starts (proposed: lobby-only edits).
  • Chat scope & safety. MVP is plain broadcast to everyone in the game (no whispers, no cross-game channels), capped at 280 chars with light rate-limiting. Open questions: profanity filtering, message retention/clearing, and whether to keep chat after the game finishes (proposed: keep, so the results screen stays social).
  • Finishing (§8). Confirm: spectating is opt-in from the score screen; a voluntary abandon action ends a party in place. Open questions — what an abandoned/wiped party's cards do (proposed: left in place as lootable contents, true to the board game), and whether a long-idle active seat is auto-abandoned by a turn timeout (host-skippable). Combined results post per-party to the leaderboard, tagged multiplayer.
Bottom line Phases 1 & 4 deliver the requirement — create, share a code, join from another browser, take turns (including party selection) and play one shared cave to a ranked finish. Phases 5–6 add the rulebook's full competitive depth and can follow once the core is proven.
